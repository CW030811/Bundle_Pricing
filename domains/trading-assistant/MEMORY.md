# trading-assistant / MEMORY

## 稳定约束

- 默认纸交易，不默认实盘。
- 实盘必须显式开启配置、传入 `--confirm-live`、具备 OKX 凭证并通过风控。
- 不需要、也不应申请提现权限。
- 风控优先于收益目标。
- 每个信号、订单和风控拦截必须可审计。

## 长期目标

- 构建稳定、可审计的 BTC/ETH 全流程量化系统：数据、策略、回测、纸交易、风控、受控自动交易、复盘。
- 把量化策略开发理解为“验证可交易假设”，不是寻找漂亮收益曲线。
- 长期数据底座采用 ODS / DWD / DWS / ADS 分层：原始数据可回溯，标准化数据可复用，因子可版本化，策略服务数据可审计。

## 策略研究原则

- 每个策略先有 Strategy Card：来源、核心假设、数据要求、进出场、仓位、风控、失效条件、复现难度。
- 策略逻辑优先于回测曲线；无法解释收益来源、市场机制和失效条件的策略不进入 paper/live。
- 简单、清楚、可解释的策略优先；复杂模型必须先解决数据、标签、泄露、成本和样本外验证。
- 回测必须使用已确认 K 线、避免未来函数，并明确手续费、滑点、资金费率和成交假设。
- 策略至少和 BTC buy-and-hold、空仓、简单趋势/动量、随机入场等基准比较。
- 参数选择必须看稳定区域、样本内/样本外、walk-forward、市场状态分解和成本敏感性，不只看最优点。
- 论坛、社交媒体和收益截图只能作为线索；可信结论来自可复现数据、扣成本回测、样本外和 paper trading。

## 数据平台原则

- CSV/Parquet 可以作为本地 DWD 文件层，但不能成为唯一长期治理方式；采集、质量、因子和策略结果要进入可查询审计库。
- 原始 OKX/API 返回先写 ODS，不直接覆盖；清洗错误、字段变化或数据源变化时必须能回溯。
- 字段命名统一使用 crypto 语义：`exchange`、`symbol`、`market_type`、`interval`、`timestamp/ts`、`open/high/low/close/volume`。
- 每次采集记录 ingestion log，每次入库后记录数据质量问题：重复、缺口、空值、异常价格、负成交量、OHLC 关系错误、未闭合 K 线。
- 回测和策略原则上读取 DWD/DWS/ADS，不直接依赖 ODS 原始结构。

## 关键风控原则

- 单笔风险控制
- 单品种暴露限制
- 日内止损阈值
- 连续亏损降级/暂停
- kill switch 可立即阻止新订单
