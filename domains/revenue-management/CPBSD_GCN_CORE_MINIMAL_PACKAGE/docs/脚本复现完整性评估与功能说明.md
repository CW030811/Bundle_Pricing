# CPBSD-GCN 核心脚本复现完整性评估与功能说明

本文档面向论文审查打包场景，说明 `CPBSD_GCN_CORE_MINIMAL_PACKAGE` 中 12 个入口脚本的作用、预期产物、已有结果日志，以及当前复现完整性边界。

## 一、结论

当前 12 个脚本**不能单独完整复现目前所有结果**。

它们目前更准确的定位是：

- CPBSD-GCN 方向的核心实验入口索引；
- 每个实验环节的最小代表性 driver；
- 配套少量代表性日志/汇总结果，用于快速审阅实验脉络；
- 原项目完整实验体系的蒸馏包，而不是独立、可直接交给审稿人运行的完整 replication package。

如果在原始项目目录结构、原始虚拟环境、Gurobi 授权、PyTorch/PyG 环境、所有 helper 模块和历史生成数据都完整存在的情况下，这些脚本可以覆盖主要实验环节的重跑路径。但如果把当前目录单独打包给审稿人，脚本会因为 helper 模块缺失、绝对路径、环境未锁定、原始数据/模型 checkpoint 未完整随包等问题，无法保证独立复现。

## 二、当前包可以证明什么

当前包适合证明：

1. 我们围绕 CPBSD-GCN 做过哪些核心实验环节。
2. 每个环节对应哪个可复用入口脚本。
3. 每个环节已有代表性日志或汇总结果。
4. 实验脉络从原 GCN 复现、CPBSD 数据生成、MB 标签、GCN 训练、GCN 效果评估，一直延伸到随机成本泛化、Phase 2 对比和 Phase 3 问题诊断。

当前包不应声明为：

1. 完整论文复现包。
2. 可在任意新机器上直接一键复现实验结果的 standalone package。
3. 覆盖所有历史结果文件、所有实例、所有 per-seed/per-setting 细节日志的全集归档。

## 三、若要成为论文审查完整包，需要补齐的内容

建议在提交论文审查前补齐以下内容：

1. **依赖环境锁定**
   - `requirements.txt` 或 `environment.yml`
   - Python 版本
   - PyTorch、torch-geometric、numpy、pandas、msgpack、gurobipy 等版本
   - Gurobi 版本、授权要求和求解参数说明

2. **helper 模块纳入或安装说明**
   当前 12 个脚本会依赖原项目中的低层模块，例如：
   - `generate_data_CPBSD.py`
   - `solve_cpbsd_milp.py`
   - `solve_cpbsd_a.py`
   - `solve_mb_bsp_on_cpbsd_v2.py`
   - `Training_multi_layer_cpbsd_mb_x.py`
   - `run_cpbsd_fcp_pruned_mb_compare_parallel.py`
   - `run_phase3_oos_bsp_completion_probe.py`

3. **路径参数化**
   多数脚本仍带有本机绝对路径，例如 `/Users/sensen/.openclaw/workspace/...`。审稿包应改成相对路径、命令行参数或环境变量。

4. **数据与模型产物策略**
   需要明确是：
   - 从随机种子重新生成全部数据；
   - 还是随包提供固定数据与 checkpoint；
   - 或二者都提供，并用 checksum 验证。

5. **一键复现入口**
   建议增加：
   - `run_all_smoke.sh`：审稿人可在较短时间内验证流程；
   - `run_all_full.sh`：完整重跑实验；
   - `VERIFY_RESULTS.md`：说明哪些数值应该一致、哪些因 MIP 求解器/硬件可能有浮动。

6. **结果完整性说明**
   当前包只保留代表性日志和汇总，不包含所有 raw instances、per-instance JSON、plots、chunk labels。论文审查包应说明这些文件为何排除，或提供外部归档链接。

## 四、12 个入口脚本功能与预期结果

### 01_original_gcn_reproduction_report.py

**实验环节**：原始 GCN reproduction/reporting。

**预期功能**：

- 汇总原 GCN/Top-K local search 相关实验结果；
- 读取已有 CSV 结果；
- 生成策略对比报告；
- 用于复现早期 GCN 加速思想在 MB/BSP/FCP 代码框架中的整理结果。

**预期结果**：

- 生成或更新原始 GCN reproduction 的对比报告；
- 对比原 local search、global top-k、不同 `K` 设定下的 revenue ratio 和 runtime ratio。

**当前包内代表性结果**：

- `results/01_original_reproduction/CPBSD_GCN_ACCELERATION_STATUS.md`

**完整性判断**：

- 不是独立完整复现脚本。它依赖原始 reproduction 结果 CSV 和原始 Mac 目录布局。

### 02_cpbsd_baselines_v2_driver.py

**实验环节**：CPBSD baseline/data setup 与基线求解。

**预期功能**：

- 生成 CPBSD baseline 实验实例；
- 调用 CPBSD-MILP、CPBSD-A、BSP、MB 等基线求解器；
- 统一记录 in-sample / out-of-sample revenue；
- 输出统一日志和每个 baseline 的 JSON 结果。

**预期结果**：

- `unified_log.csv`
- `unified_log.json`
- `results/baseline_*.json`
- baseline 汇总报告。

**当前包内代表性结果**：

- `results/02_cpbsd_baselines/unified_log.csv`
- `results/02_cpbsd_baselines/Baseline_V2_Results_Summary.md`

**完整性判断**：

- 作为入口脚本是完整的；
- 作为 standalone 审查脚本不完整，因为依赖 `generate_data_CPBSD.py`、各类 solver、Gurobi 环境和原始路径。

### 03_gcn_dataset_setup.py

**实验环节**：GCN training data setup。

**预期功能**：

- 按指定 CPBSD setting 生成一批实例；
- 将实例划分为 train/eval/test；
- 生成 manifest CSV/JSON 和 dataset summary。

**预期结果**：

- `dataset_summary.json`
- `train_manifest.csv/json`
- `eval_manifest.csv/json`
- `test_manifest.csv/json`
- 可选 train/eval/test 子目录实例拷贝。

**当前包内代表性结果**：

- `results/03_gcn_dataset_setup/dataset_summary.json`

**完整性判断**：

- 入口逻辑清楚；
- 但当前包没有随附 raw `.msgpack` instances，也没有包含生成函数模块，因此不能单独完成数据生成。

### 04_mb_label_chunk_queue.py

**实验环节**：MB label generation。

**预期功能**：

- 对 train/eval/test manifests 进行 chunk 化队列求解；
- 调用 MB solver 为每个 CPBSD 实例生成标签；
- 记录 queue status、chunk status 和求解日志。

**预期结果**：

- `full_queue_status.json`
- `train_queue_status.json`
- `eval_queue_status.json`
- `test_queue_status.json`
- 每个 chunk 的 MB 结果 CSV/JSON；
- MB label generation 执行日志。

**当前包内代表性结果**：

- `results/04_mb_label_generation/full_queue_status.json`
- `results/04_mb_label_generation/cpbsd_mb_labels_q.log`

**完整性判断**：

- 作为队列入口脚本可复用；
- 当前包未包含完整 chunk manifests、raw instances 和 MB solver 依赖，不能单独重建全部标签。

### 05_gcn_mb_training_pipeline.py

**实验环节**：GCN model training。

**预期功能**：

- 合并 MB label 结果；
- 执行 smoke training；
- 执行 full training；
- 自动处理 MPS/CPU fallback；
- 输出模型、loss 曲线、metrics 和训练日志。

**预期结果**：

- `metrics_edge_cpbsd_mb_x_2layer_seed42.json`
- `train_loss_*.csv`
- `val_loss_*.csv`
- 模型 checkpoint；
- `tmux_train_run.log`

**当前包内代表性结果**：

- `results/05_gcn_training/tmux_train_run.log`
- `results/05_gcn_training/metrics_edge_cpbsd_mb_x_2layer_seed42.json`

**完整性判断**：

- pipeline 入口完整；
- 但当前包没有包含训练用 merged CSV、PyTorch/PyG 依赖锁定和训练核心模型定义模块的完整独立拷贝。

### 06_gcn_fcp_pruned_mb_compare.py

**实验环节**：GCN effect evaluation。

**预期功能**：

- 载入训练好的 GCN 模型；
- 对 CPBSD 实例构建图并生成 edge probability；

- 比较 FCP-pruned-MB、BSP、CPBSD-A 等方法；
- 输出 per-instance 和 aggregate comparison。

**预期结果**：

- `aggregate_metrics.json`
- `comparison_summary_all.csv`
- 每个 seed/run 的 `comparison_summary.json/csv`

**当前包内代表性结果**：

- `results/06_gcn_effect_eval/aggregate_metrics.json`
- `results/06_gcn_effect_eval/comparison_summary_all.csv`

**完整性判断**：

- 是 GCN 效果实验的核心入口；
- 但当前包未包含模型 checkpoint、实例数据、solver helper 和完整随机种子运行目录。

**当前包内代表性结果**：

## 五、建议给论文审查包的命名与表述

如果当前不继续补 helper 和环境，建议对外命名为：

> CPBSD-GCN Core Artifact Index and Representative Logs

不要命名为：

> Full Reproduction Package

更稳妥的说明是：

> This package contains the distilled entry scripts and representative logs for the core CPBSD-GCN experimental pipeline. Full end-to-end reproduction requires the original project helper modules, solver environment, generated instances, and Gurobi/PyTorch dependencies documented in the manifest.

## 六、推荐下一步

若目标是论文审查可运行包，建议新增一个 `REPRODUCIBILITY_PACKAGE/`，内容包括：

1. `requirements.txt` 或 `environment.yml`
2. `run_all_smoke.sh`
3. `run_all_full.sh`
4. `configs/*.yaml`
5. `src/` 中完整 helper modules
6. `data/README.md` 说明 raw data 是随包提供还是由脚本生成
7. `EXPECTED_RESULTS.md` 记录每个实验应得到的关键数值和允许误差
8. `CHECKSUMS.txt` 记录固定结果文件校验值

当前 `CPBSD_GCN_CORE_MINIMAL_PACKAGE` 可以作为这个完整包的骨架和索引来源。
